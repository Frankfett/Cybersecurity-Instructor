Introduction

Vulnerability assessment is the systematic process of identifying, quantifying, and prioritizing (or ranking) the vulnerabilities in systems. It's one of the most important activities in cybersecurity, helping organizations understand their security posture and guide remediation efforts.

A vulnerability assessment differs from a penetration test:

  • Vulnerability Assessment: Identifies vulnerabilities that could be exploited (automated scanning)
  • Penetration Test: Actually exploits vulnerabilities to demonstrate impact (manual testing)

⚠️ Important Legal Notice

Only perform vulnerability assessments on systems you own or have explicit written permission to test. Unauthorized scanning is illegal and may violate computer fraud laws. This lab uses controlled lab environments provided by CNM.

Fundamental Concepts

Vulnerability vs. Exposure vs. Risk

Understanding these terms is critical to vulnerability management:

  • Vulnerability: A weakness in software or hardware that could be exploited (e.g., unpatched software)
  • Exposure: A weakness that can be accessed by an attacker (e.g., open port running vulnerable service)
  • Risk: The potential for loss or damage if a vulnerability is exploited (Vulnerability × Likelihood × Impact)

CVSS Scoring System

The Common Vulnerability Scoring System (CVSS) provides a standardized way to rate vulnerability severity. CVSS v3.1 is the current standard.

Severity Rating CVSS Score Range Description
CRITICAL 9.0 - 10.0 Immediate action required; could lead to complete system compromise
HIGH 7.0 - 8.9 Significant risk; should be remediated within 30 days
MEDIUM 4.0 - 6.9 Moderate risk; remediate within 90 days
LOW 0.1 - 3.9 Minor issue; address in regular maintenance windows

Assessment Methodology

A typical vulnerability assessment follows this workflow:

  1. Planning & Scoping: Define assets, scope, and objectives
  2. Discovery: Identify systems, services, and configurations
  3. Scanning: Run automated vulnerability scanners
  4. Analysis: Review results and filter false positives
  5. Reporting: Document findings with severity and recommendations
  6. Remediation Tracking: Monitor fixes and re-scan to verify

Nessus Scanning Workflow

Nessus is the most widely-used vulnerability scanner. Here's the basic workflow:

# 1. Launch Nessus
sudo systemctl start nessusd

# 2. Access web interface (default port 8834)
https://localhost:8834

# 3. Create new scan policy
- Select "Vulnerabilities & Settings" template
- Specify target IP or range (e.g., 192.168.1.0/24)
- Configure discovery settings

# 4. Launch scan and monitor progress
- Allow scan to complete (may take 30+ minutes)
- Monitor for active scans

# 5. Review results
- Filter by severity (Critical, High, Medium)
- Export findings for reporting

OpenVAS Scanning

For open-source vulnerability scanning:

# Start OpenVAS services
sudo gvm-start

# Access web interface (default port 9392)
https://127.0.0.1:9392

# Create task with target
Scanning > Tasks > New Task
- Set Name and Comment
- Select Target (create if needed)
- Choose Scan Config (Full and fast recommended)

# Run and review scan
Start scan and monitor until completion

💡 Pro Tips

  • Always start with a network discovery scan to identify live hosts
  • Use credentials when possible for authenticated scans (more accurate)
  • Schedule scans during maintenance windows to avoid network impact
  • Keep scanner plugins/feeds updated for latest vulnerability data
  • Use multiple scanners to cross-validate findings

Analyzing Scan Results

Common Vulnerabilities to Understand

Vulnerability Type Description Typical Fix
Unpatched Software Outdated applications with known CVEs Apply security patches and updates
Default Credentials Services using factory default passwords Change all default usernames and passwords
Weak Encryption Old/insecure protocols (HTTP, SSLv3, weak ciphers) Enable TLS 1.2+, disable weak protocols
Open Ports Unnecessary services listening on network Close unnecessary ports and services
Missing Security Headers Web apps lacking security HTTP headers Add Content-Security-Policy, X-Frame-Options, etc.
SQL Injection Improper input validation in databases Use parameterized queries and input validation

False Positives and Verification

Vulnerability scanners often produce false positives. Always verify findings:

  • Verify Manually: Manually test the reported vulnerability
  • Check Context: Determine if vulnerability applies to this system
  • Review Patch Status: Confirm whether the system is actually vulnerable
  • Document Exceptions: Record accepted risks or false positives

Hands-On Exercises

Exercise 1: Network Discovery Scan

Objective: Identify all active hosts on the lab network.

  1. Open Nessus or OpenVAS
  2. Create a new scan with network discovery plugin only
  3. Specify lab network range (e.g., 192.168.100.0/24)
  4. Run scan and identify all active hosts
  5. Document findings: IP addresses, discovered hostnames

Exercise 2: Vulnerability Scan of Linux System

Objective: Run a full vulnerability scan on a Linux target.

  1. Select a Linux VM as target (e.g., Ubuntu or CentOS)
  2. Create new scan using "Vulnerabilities & Settings" template
  3. Configure for authenticated scan if possible
  4. Run scan and allow 30-45 minutes for completion
  5. Review results sorted by severity
  6. Identify 3-5 CRITICAL or HIGH vulnerabilities
  7. Document each finding: title, CVSS score, description

Exercise 3: Scan Analysis and False Positive Review

Objective: Analyze scan results and identify false positives.

  1. Export scan results to CSV or PDF
  2. Manually verify 5 random findings
  3. For each finding:
    • Record the vulnerability name
    • Note the reported severity
    • Manually test or verify if it's accurate
    • Document whether it's true positive or false positive
  4. Calculate false positive rate

Exercise 4: Vulnerability Report Generation

Objective: Create a professional vulnerability assessment report.

  1. Gather results from Exercise 2
  2. Create report containing:
    • Executive summary
    • Vulnerability breakdown by severity
    • Top 10 vulnerabilities with CVSS scores
    • Remediation recommendations
    • Risk assessment matrix
  3. Format professionally with charts/graphs if possible
  4. Include disclaimer about authorized testing

Exercise 5: Remediation Planning

Objective: Plan remediation based on risk prioritization.

  1. Take vulnerabilities from Exercise 2
  2. For each CRITICAL finding, document:
    • What is vulnerable
    • Why it's critical
    • Recommended fix
    • Estimated remediation time
    • Impact of fix (service disruption, dependencies)
  3. Create remediation timeline

Key Takeaways

🎯 Remember These Points

  • Vulnerability assessment is a systematic process requiring proper scoping
  • CVSS scores provide standardized severity ratings for all vulnerabilities
  • False positives are common—always verify scan findings
  • Risk is not just about vulnerability severity; consider likelihood and impact
  • Remediation prioritization should balance risk with operational feasibility
  • Regular re-scanning verifies that vulnerabilities have been fixed
  • Authorization is legally and ethically mandatory
